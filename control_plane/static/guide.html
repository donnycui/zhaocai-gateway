招财网关使用指南

💡 一句话总结

招财网关是一个 OpenAI 兼容的 AI 推理网关 + 多节点配置集中管理平台。 让你用一个统一入口管理多个 AI 提供商，并轻松管理数十个节点的模型配置。

开发背景

在实际使用 AI 模型时，我们常常遇到以下问题：

  • 多平台管理混乱 - 同时使用 OpenAI、Claude、DeepSeek、SiliconFlow 等多个提供商，每个都有独立的 API Key、端点和模型名称
  • 节点配置繁琐 - 有多台服务器（VPS、树莓派等）运行 OpenClaw，每台都要单独配置 openclaw.json，修改时需要逐台登录
  • 配置不同步 - 新增模型或更换 API Key 时，部分节点容易遗漏，导致服务异常
  • 故障无备份 - 某个 Provider 故障时，无法自动切换到备用 Provider

招财网关就是为了解决这些问题而设计的。它提供一个统一的控制面板集中管理所有 Provider， 通过配置分发机制让节点自动拉取最新配置，并支持故障自动转移保证服务可用性。

核心功能

1. 统一 AI 推理网关

  • OpenAI 兼容接口 - 提供标准 /v1/chat/completions 接口，已有代码无需修改即可接入
  • 多 Provider 路由 - 支持轮询、权重、优先级三种路由策略，自动分配请求到不同 Provider
  • 故障自动转移 - 单个 Provider 故障时自动重试其他 Provider，保证服务连续性
  • 格式自动转换 - 自动处理 OpenAI 和 Anthropic 的协议差异，对外暴露统一接口
  • 流式响应支持 - 支持 SSE 流式输出，兼容 GPT、Claude 等主流模型

2. 配置集中管理

  • Provider 管理 - 集中管理所有 AI 提供商的 API Key、端点、模型映射
  • Model 别名 - 将复杂的上游模型名（如 deepseek-ai/DeepSeek-V3）映射为简洁别名（如 deepseek-v3）
  • 配置集 (Profile) - 按场景定义模型集合，如"通用配置"、"代码助手"、"中文优化"
  • 节点分组 - 将多台服务器分组管理，不同组使用不同配置

3. 配置自动分发

  • 节点注册 - 为每个节点生成唯一的拉取 Token
  • 增量同步 - 节点通过 ETag 机制只拉取变更配置，节省带宽
  • 自动重载 - 配置变更后节点自动重载 OpenClaw，无需人工干预
  • 版本追踪 - 保存配置历史版本，支持回滚

4. 运维监控

  • 健康检查 - 实时监控所有 Provider 可用性和响应延迟
  • 配置审计 - 记录所有配置变更，追踪谁在何时修改了什么
  • Token 轮换 - 支持节点 Token 动态轮换，提升安全性

适用场景

场景 说明
个人多节点部署 家里树莓派 + VPS 组合，统一管理配置
团队共享网关 团队共用一个网关入口，统一配额管理
多模型 A/B 测试 快速切换不同模型对比效果
故障自动切换 主 Provider 故障时自动切换到备用

系统架构

招财网关采用 中心控制面 + 节点拉取 的架构模式：

  • 中心控制面 - 统一维护 Provider/Model/Profile/Node，编译生成节点专属配置
  • 节点 - 通过 Token 定时拉取配置，基于 ETag 判断是否有更新
  • 推理网关 - 提供 OpenAI 兼容接口，支持多 Provider 路由和故障转移

快速开始

1
安装依赖
pip install -r requirements.txt
2
配置环境变量
cp .env.example .env

# 编辑 .env，设置以下必需项：
ZHAOCAI_ENCRYPTION_KEY="生成加密密钥"
ZHAOCAI_ADMIN_TOKEN="设置管理员Token"
SILICONFLOW_API_KEY="你的API Key"
3
启动服务
python gateway.py

服务启动后访问：

  • 根接口（JSON，不是首页）: http://localhost:8000/
  • 使用指南（本页）: http://localhost:8000/guide
  • 健康 API（JSON）: http://localhost:8000/health
  • 健康监控页（可视化）: http://localhost:8000/health-ui
  • 控制面板: http://localhost:8000/control
  • API 文档 / Swagger 调试台: http://localhost:8000/docs
入口说明

/ 和 /health 返回的是 JSON，所以浏览器里通常只会显示一行代码；这是正常现象。

如果你想看可视化页面，请优先访问 /guide、/health-ui 和 /control。

系统架构

招财网关采用 中心控制面 + 节点拉取 的架构模式：

  • 中心控制面：统一维护 Provider/Model/Profile/Node，编译生成节点专属配置
  • 节点：通过 Token 定时拉取配置，基于 ETag 判断是否有更新
  • 推理网关：提供 OpenAI 兼容接口，支持多 Provider 路由和故障转移

什么是 Provider？

Provider 是上游 AI 服务提供商的配置，例如 OpenAI、SiliconFlow、Anthropic 等。

每个 Provider 包含：

字段 说明 示例
name 唯一标识名称 siliconflow-main
provider_type 类型（openai/anthropic） openai
base_url API 基础地址 https://api.siliconflow.cn/v1
api_key 认证密钥（加密存储） sk-xxx
auth_scheme 认证方式 bearer / x-api-key

什么是 Model？

Model 是模型的别名配置，将上游模型名称映射为节点使用的友好名称。

💡 为什么需要 Model？

不同 Provider 可能有相同的模型名称（如都有 gpt-4o），通过 Model 别名可以区分来源，同时隐藏真实的上游 Provider 细节。

什么是 Profile？

Profile（配置集） 是一组模型的集合，用于定义某个场景下可用的模型。

例如：

  • 默认配置：通用场景，包含 gpt-4o、claude-3-sonnet
  • 代码助手：编程场景，包含 deepseek-coder、claude-3-opus
  • 中文优化：中文场景，包含 qwen-max、ERNIE-4

什么是 Node？

Node（节点） 代表一个 OpenClaw 实例，需要关联到一个 Profile。

每个 Node 拥有：

  • 唯一的拉取 Token（用于身份验证）
  • 关联的 Profile（决定可用模型）
  • 专属的 openclaw.json 配置

完整工作流程

1
创建 Provider

在控制面板或调用 API 创建上游 Provider 配置。

POST /control/v1/providers
{
  "name": "siliconflow",
  "provider_type": "openai",
  "base_url": "https://api.siliconflow.cn/v1",
  "api_key": "sk-xxx"
}
2
创建 Model

为 Provider 创建模型别名。

POST /control/v1/models
{
  "provider_id": 1,
  "upstream_model": "deepseek-ai/DeepSeek-V3",
  "alias": "deepseek-v3"
}
3
创建 Profile 并绑定模型
POST /control/v1/profiles
{ "name": "默认配置", "description": "通用场景" }

POST /control/v1/profiles/1/bindings
{ "model_ids": [1, 2, 3] }
4
创建 Node
POST /control/v1/nodes
{
  "name": "节点-1",
  "profile_id": 1,
  "sync_mode": "pull"
}

创建成功后会返回 pull_token，用于节点拉取配置。

5
节点拉取配置
GET /control/v1/nodes/1/openclaw-json
Authorization: Bearer {pull_token}

节点保存配置后重启 OpenClaw 即可生效。

API 使用示例

推理接口

# 非流式请求
curl http://localhost:8000/v1/chat/completions \
  -H "Content-Type: application/json" \
  -d '{
    "model": "deepseek-v3",
    "messages": [{"role":"user","content":"你好"}]
  }'

# 流式请求
curl http://localhost:8000/v1/chat/completions \
  -H "Content-Type: application/json" \
  -d '{
    "model": "deepseek-v3",
    "messages": [{"role":"user","content":"你好"}],
    "stream": true
  }'

节点同步代理

使用内置脚本自动同步配置：

python scripts/node_sync_agent.py \
  --base-url http://127.0.0.1:8000 \
  --node-id 1 \
  --pull-token zg_node_1_xxx \
  --output /etc/openclaw/openclaw.json \
  --interval 60 \
  --reload-cmd "systemctl restart openclaw"

环境变量参考

变量 必需 说明
ZHAOCAI_ENCRYPTION_KEY ✅ API Key 加密密钥（Fernet 格式）
ZHAOCAI_ADMIN_TOKEN ✅ 管理接口认证 Token

常见问题

Q: 启动报错 "ZHAOCAI_ENCRYPTION_KEY not set"

生成密钥并添加到 .env：

python -c "from cryptography.fernet import Fernet; print(Fernet.generate_key().decode())"

Q: Provider 健康检查失败（401）

这是正常的，表示 API Key 未配置或无效。在 .env 中配置正确的 API Key 后重启服务。

Q: 如何快速初始化 Provider？

使用 bootstrap 脚本：

python scripts/bootstrap_provider.py \
  --base-url http://127.0.0.1:8000 \
  --admin-token "$ZHAOCAI_ADMIN_TOKEN" \
  --name siliconflow \
  --api-key "sk-xxx" \
  --models "deepseek-ai/DeepSeek-V3,deepseek-ai/DeepSeek-R1"
